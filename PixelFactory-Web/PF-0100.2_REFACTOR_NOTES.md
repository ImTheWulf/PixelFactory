# PF-0100.2 — app.js Split Into Load-Ordered Files

## What changed

`frontend/static/app.js` (4,224 lines, everything in one file, all global
scope) is now 5 files under `frontend/static/js/`:

| File | Lines | Contains |
|---|---|---|
| `00_shared-init.js` | 110 | Nav bar, view switching, ComfyUI status check, Reload UI button |
| `01_palette-lab.js` | 2,682 | Palette Lab: pixel snap, compare viewer, all cleanup toggles, save/export flow |
| `02_generation-studios.js` | 316 | Character Studio generation + Tile Studio generation |
| `03_asset-browser-export.js` | 844 | Asset Browser, selection, Godot/Aseprite export panel |
| `04_image-viewer.js` | 272 | Fullscreen image viewer modal |

`index.html` now loads all 5 in that exact order via 5 `<script>` tags,
keeping your existing `?v=pf0044_palette_lab_v2` cache-busting string.

## Why the split is where it is (and why Palette Lab is still huge)

I cut at the natural section boundaries already in your code — every
section starts with its own block of `const x = document.getElementById(...)`
declarations right before the functions that use them. I split exactly at
those boundaries, in the original order, with nothing moved or reordered.

That means `01_palette-lab.js` is still 2,682 lines. This pass fixes "one
4,224-line unsearchable file" — it does not fix "Palette Lab itself has a
lot of surface area." That's a real, separate problem (worth tackling later
by pulling pixel-snap-panel logic, compare-viewer logic, and
save/history logic into their own files), but doing that requires tracing
which functions call which across sections, which is a different — riskier
— kind of change than this one. I didn't want to bundle that risk into a
"just reorganize" step.

## What did NOT change

- No function body, variable, or event listener logic changed.
- No reordering — file 2 runs exactly where the old code used to run in
  the middle of the original file, just now in its own file.
- No backend changes.

## How this was verified (and the one thing I could NOT verify)

**Proven:** concatenating the 5 new files back together in load order
reproduces the original `app.js` byte-for-byte — identical MD5 hash. That
means nothing was lost, duplicated, or reordered, full stop.

**Proven:** each of the 5 files passes `node --check` on its own (valid,
parseable JavaScript by itself). If a boundary had cut through the middle
of a function or object literal, this would have caught it immediately —
none did.

**NOT verified:** actual execution in a browser. I don't have a headless
browser/DOM here, so unlike the backend refactor (which I regression-tested
against real images), I can't hand you a "ran it and confirmed identical
output" receipt for this one. The two checks above are strong — they
guarantee the code is intact and unchanged — but they can't substitute for
you clicking through the app once.

## How to test

1. Drop `frontend/static/js/` (5 files) and the updated
   `frontend/templates/index.html` into your project.
2. Restart the server, hard-refresh the page (or bump the `?v=` string if
   your browser caches aggressively).
3. Open the browser console (F12) and check for any red errors on load —
   if a boundary were somehow wrong, this is where it'd show immediately as
   an "X is not defined" error.
4. Click through: switch views (nav bar), check ComfyUI status, load an
   image into Palette Lab and run a repair pass, generate a character,
   generate a tile, open Asset Browser and select/export something, open
   the fullscreen image viewer.
5. Once confirmed, delete or rename the old `frontend/static/app.js` — it's
   no longer referenced by `index.html`, so leaving it there is just
   clutter, not a risk.

## If something breaks

Paste whatever shows up in the browser console (F12 → Console tab) — for a
split like this, a break is almost always an immediate, loud "X is not
defined" on page load, not a subtle behavioral difference. That'll point
straight at which file/boundary needs a look.

## Next step (only after you confirm this one is good)

If you want real encapsulation later (ES modules, explicit
imports/exports, no more implicit globals) — that's a legitimate follow-up,
but it's a different kind of change than this one and deserves its own
pass with more hands-on testing, not a bundle-in.
